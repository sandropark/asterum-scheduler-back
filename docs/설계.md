# 일정
  - 제목
  - 시작일시
  - 종료일시
  - 삭제일시
  - 반복규칙 (rrule)
    - 옵션: 매일, 매주, 매월
    - 종료 조건: 특정 날짜까지 반복 / 특정 횟수 반복 / 무기한 반복

# 테이블
- **events**:
  - **컬럼**: 제목, 시작, 종료일시, 삭제일시, 반복규칙, location_id(nullable FK)
  - 일정을 기록
- **events_instances**:
  - **컬럼**: eventId, 제목(nullable), 시작, 종료일시, 삭제일시, has_override_participants(bool), has_override_location(bool), location_id(nullable FK), status(enum)
  - 반복 일정의 각 인스턴스를 기록.
  - 제목이 null 이면 events.제목 을 따름. 값 있으면 인스턴스별 오버라이드.
  - has_override_participants = true 면 instance_participants 를 참조, false 면 event_participants 를 참조.
  - has_override_location = true 면 instance.location_id 사용 (null 이면 이 회차만 장소 없음, 값 있으면 다른 장소), false 면 event.location_id 따름.
  - status: CONFIRMED | CONFLICT. 같은 장소 + 시간 겹침 시 CONFLICT.
  - 정책으로 정해놓은 최대 기간이 있고(예: 10년) 그 기간내의 일정은 모두 생성되어 있다.
  - 최대 기간을 넘어가는 반복 일정들이 있기 때문에 **스케줄러**를 돌려서 인스턴스를 생성한다.


- **users**:
  - **컬럼**: id, 이름, email, is_team(bool)
  - 사용자 정보. is_team = true 이면 팀 (자녀 유저들의 부모).
- **user_memberships**:
  - **컬럼**: team_user_id, member_user_id
  - 팀-멤버 N:M 관계 (2단계). team_user_id 는 is_team=true, member_user_id 는 is_team=false.
- **event_participants**:
  - **컬럼**: eventId, userId
  - 일정의 시리즈 기본 참여자.
- **instance_participants**:
  - **컬럼**: instanceId, userId
  - 인스턴스별 오버라이드 참여자. has_override_participants = true 일 때만 참조됨.


- **locations**:
  - **컬럼**: id, 이름
  - 장소 마스터.

# 유즈케이스

## Phase A

- ### 월간 일정 조회
  - **응답**: 제목, 시작, 종료일시, 장소, status
  - 시작, 종료일시(기본값 당월)를 파라미터로 받아서 **events_instances** 테이블에서 데이터 조회
  - 삭제일시 == null 인 데이터만 조회
  - 제목은 instance.제목 이 있으면 그 값, null 이면 events.제목.
  - 장소는 has_override_location = true 면 instance.location_id, false 면 events.location_id.

- ### 일정 상세 조회
  - **입력**: instance.id
  - **응답**: 제목, 시작, 종료일시, 반복규칙, 참여자, 장소, status
  - events_instances 에서 id 로 조회. 삭제일시 == null 인 데이터만.
  - 제목은 instance.제목 이 있으면 그 값, null 이면 events.제목.
  - 반복규칙은 events.반복규칙 (단일 일정이면 null).
  - 참여자는 has_override_participants 에 따라 event_participants 또는 instance_participants 에서 조회. 팀 userId 가 포함된 경우 user_memberships 조인해 멤버 userId 들도 함께 반환.
  - 장소는 has_override_location = true 면 instance.location_id, false 면 events.location_id.

- ### 단일 일정 생성
  - events(제목, 시작, 종료일시) 생성
  - events_instances(eventId, 시작, 종료일시) 생성 (제목은 null — events 의 제목 사용)

- ### 단일 일정 수정
  - 제목
    - events 제목만 수정
  - 시간
    - events 시간 수정, events_instances 시간 수정

- ### 단일 일정 삭제
  - events, events_instances 삭제일시 추가

- ### 반복 일정 생성
  - events(제목, 시작, 종료일시, 반복규칙) 생성.
  - 반복 규칙에 따라서 events_instances(eventId, 시작, 종료일시) 생성

- ### **반복 일정 수정/삭제**
  - 반복 일정 중에서 수정/삭제하는 경우 사용자에게 3가지 선택지가 주어진다.
  1. 이 일정만
  2. 이 일정 및 향후 일정만
  3. 전체 일정

- ### 반복 일정 수정
  - **이 일정**
    - 해당 instance 의 필드(제목/시간)만 직접 수정
  - **이 일정 및 향후 일정**
    - 기존 events 반복규칙 종료 조건을 "이 일정" 전날로 단축
    - "이 일정" 이후를 포함하는 새로운 events 생성 (수정된 필드 반영, 종료조건 승계: 특정 날짜까지 → 그대로 / 무기한 → 무기한 / 특정 횟수 → 차감)
    - 제목 수정: 해당 eventId 의 "이 일정" 이후의 instance 의 eventId 를 신규 eventId 로 갱신, instance.제목 을 null 로 리셋 (오버라이드도 함께 덮어쓰기)
    - 시간/반복규칙 수정: 해당 eventId의 "이 일정"이후의 instance 삭제 후 신규 events 기준으로 재생성. 이때 deleted_at이 not null인 instance는 갱신되지 않음.
  - **전체 일정**
    - 제목 수정: events.제목 수정, 해당 eventId의 instance의 title을 null로 변경. (단일 수정했던 일정들까지 함께 수정)
    - 시간/반복규칙 수정: events 수정 후 해당 eventId 의 모든 instances 삭제 후 재생성. 이때 deleted_at이 not null인 instance는 갱신되지 않음.

- ### 반복 일정 삭제
  - **이 일정**
    - 해당 instance 의 삭제일시 설정
  - **이 일정 및 향후 일정**
    - 기존 events 반복규칙 종료 조건을 "이 일정" 전날로 단축
    - 기존 eventId 의 "이 일정" 이후의 instances 에 삭제일시 설정
  - **전체 일정**
    - events 삭제일시 추가
    - 해당 eventId 의 모든 events_instances 에 삭제일시 설정

## Phase B

- ### 일정에 사용자 설정, 수정
  - **참여자 조회**
    - has_override_participants = false → event_participants 의 userId 들
    - has_override_participants = true → instance_participants 의 userId 들
    - userId 중 팀 user 가 있으면 user_memberships 조인해 멤버 userId 들도 함께 반환
  - **이 일정** (반복 일정만 해당)
    - event_participants 의 userId 들을 instance_participants 로 복사
    - 추가/제외 반영
    - has_override_participants = true 로 설정
  - **이 일정 및 향후** (반복 일정만 해당)
    - 신규 event 생성 플로우 수행 + 신규 event_participants 에 반영
    - "이 일정" 이후의 instance 의 has_override_participants = false 로 리셋, instance_participants clear (deleted_at not null 인 instance 는 제외)
  - **전체**
    - event_participants 수정
    - 해당 eventId 의 모든 instance 의 has_override_participants = false 로 리셋, instance_participants clear (deleted_at not null 인 instance 는 제외)

- ### 단일 일정 -> 반복 일정 수정
  - events 에 반복규칙 추가 (다른 필드 수정도 함께 가능)
  - 해당 eventId 의 기존 instance 삭제 후 반복규칙 기준으로 재생성

- ### 일정에 팀 설정, 수정
  - 팀은 일반 user 와 동일하게 취급 (user 도메인: users + user_memberships 로 N:M 관계, 2단계).
  - event_participants / instance_participants 에는 팀 userId 만 저장. 조회 시 user_memberships 조인해 멤버 userId 들도 함께 resolve (동적 — 팀 멤버 변경 시 기존 일정에 자동 반영).
  - 팀 추가/제거, scope 별 수정 플로우는 "일정에 사용자 설정, 수정" 과 동일 (userId 단위 처리).

- ### 일정에 장소 설정, 수정
  - 일정당 0~1 장소.
  - has_override_location = false → event.location_id 따름.
  - has_override_location = true → instance.location_id 사용 (null 이면 이 회차만 장소 없음, 값 있으면 다른 장소).
  - 중복 정책 (soft-conflict): 같은 장소 + 시간 overlap 인스턴스가 있으면 새 instance.status = CONFLICT, 아니면 CONFIRMED. 기존 인스턴스는 건드리지 않음.
  - race 방지: CONFIRMED + 삭제 안 된 인스턴스에만 적용되는 **partial exclusion constraint** 로 DB 레벨 원자성 보장. INSERT 는 `status=CONFIRMED` 로 먼저 시도 → ExclusionViolation 예외 catch 시 `CONFLICT` 로 재시도 (사전 SELECT 불필요).
    - 제약: `EXCLUDE USING gist (location_id WITH =, tsrange(시작, 종료일시, '[)') WITH &&) WHERE (status = 'CONFIRMED' AND 삭제일시 IS NULL)`
  - **이 일정** (반복 일정만 해당)
    - instance.location_id 설정, has_override_location = true
    - 충돌 검사 후 instance.status 세팅
  - **이 일정 및 향후** (반복 일정만 해당)
    - 신규 event 생성 플로우 수행 + 신규 event.location_id 에 반영
    - "이 일정" 이후 instance 의 has_override_location = false 로 리셋, location_id 는 null (오버라이드 해제). deleted_at not null 인 instance 는 제외.
    - 각 instance 의 status 재검사
  - **전체**
    - event.location_id 수정
    - 해당 eventId 의 모든 instance 의 has_override_location = false 로 리셋, location_id 는 null (오버라이드 해제). deleted_at not null 인 instance 는 제외.
    - 각 instance 의 status 재검사

# 작업 순서

small step 단위로 진행. 각 step 은 독립적으로 빌드/테스트 가능해야 한다.

## Phase A — 기본 CRUD (제목·시간·반복)

> Phase A 가 끝나면 제목·시간·반복만 있는 심플 캘린더가 동작한다. 참여자·장소·팀은 다루지 않는다.

- [x] **A-1. 스키마**: `events` 테이블 신규 생성 (기본 필드: 제목, 시작, 종료, 삭제일시, 반복규칙 nullable). `events_instances` 테이블 신규 생성 (기본 필드: eventId, 제목 nullable, 시작, 종료, 삭제일시).
- [x] **A-2. 엔티티 + 단일 일정 생성** — `BaseEntity` (common, 감사 컬럼) 도입, `Event` / `EventInstance` 엔티티 생성. events + events_instances 동시 생성.
- [x] **A-3. 월간 일정 조회** — events_instances 기반. 제목 coalesce.
- [x] **A-4. 일정 상세 조회** (응답: 제목·시간·반복규칙).
- [x] **A-5. 단일 일정 수정/삭제** — events + events_instances 동기화.
- [x] **A-6. 반복 일정 생성** (rrule 라이브러리 도입, 정책 기간 내 인스턴스 일괄 전개). ⚠ 인스턴스 insert 는 **row 별 개별 insert** 로 구현 (벌크 insert 금지) — Phase B-6 의 CONFIRMED/CONFLICT 예외 재시도 패턴과 호환되도록 미리 구조 확보.
- [x] **A-7. 반복 일정 삭제 — `THIS_ONLY`** — 해당 instance 의 deletedAt 만 세팅. event 는 건드리지 않음.
- [x] **A-8. 반복 일정 삭제 — `ALL`** — event.deletedAt + 해당 eventId 의 모든 instance.deletedAt 을 동일 시점으로 세팅.
- [x] **A-9. 반복 일정 삭제 — `THIS_AND_FUTURE`** — 2 sub-step 분할.
  - [x] **A-9a. RruleShortener 구현** — rrule 종료 조건 (`UNTIL` / `COUNT` / 무기한) 을 주어진 시점으로 단축하는 순수 컴포넌트. Spring/DB 미의존, 유닛 테스트로 3 케이스 커버.
  - [x] **A-9b. EventService 통합** — `deleteThisAndFuture(instanceId)` 추가. target 포함 이후 인스턴스 soft-delete + `event.rrule` 을 `target.startAt.minusSeconds(1)` 로 단축 (A-9a 위임). `rrule == null` 이면 `INVALID_INPUT`.
- [x] **A-10. 반복 일정 수정 — `THIS_ONLY`** — 해당 instance 의 title/startAt/endAt 만 직접 오버라이드. event 와 다른 instance 는 건드리지 않음.
- [ ] **A-11. 반복 일정 수정 — `THIS_AND_FUTURE` (제목)**.
- [ ] **A-12. 반복 일정 수정 — `THIS_AND_FUTURE` (시간/반복규칙)**.
- [ ] **A-13. 반복 일정 수정 — `ALL` (제목)**.
- [ ] **A-14. 반복 일정 수정 — `ALL` (시간/반복규칙)**.
- [ ] **A-15. 스케줄러** — 배치로 정책 기간 경계 확장.
- [ ] **A-16. Controller 계층** — 각 유즈케이스별 REST 엔드포인트 추가. Request DTO 에 `@Valid` + Jakarta Bean Validation (`@NotBlank`, `@NotNull`, `@Future` 등) 적용. `GlobalExceptionHandler` 에서 `MethodArgumentNotValidException` → `ErrorCode.INVALID_INPUT` 매핑.

> **검토 예정 (별도 step)**: 현재 soft-delete 는 `@SQLRestriction("deleted_at IS NULL")` + 수동 `deletedAt` 세팅 패턴. Hibernate 6.4+ 의 `@SoftDelete(columnName = "deleted_at", strategy = TIMESTAMP)` 로 전면 교체 시 ①`em.remove` 및 derived `deleteAllBy...` / JPQL `DELETE` 가 자동으로 bulk UPDATE 로 변환됨, ②SELECT 필터 자동 주입, ③수동 `deletedAt` 세팅 삭제. 트레이드오프: `event.deletedAt == instance.deletedAt` 동일 시점 계약 (현재 `deleteAll`/`deleteSingle`) 이 DB `now()` 기반으로 바뀌며, 감사 목적으로 삭제 시점을 앱 코드에서 읽는 방식이 어색해짐. Phase A 완료 후 또는 A-16 전후에 별도 리팩토링 step 으로 재평가.

## Phase B — 참여자·팀·장소·전환

- [ ] **B-1. 스키마 확장 (참여자)**: `users` (+`is_team`) / `user_memberships` 신규. `event_participants` / `instance_participants` 신규. `events_instances` 에 `has_override_participants` 컬럼 추가.
- [ ] **B-2. 일정에 사용자 설정 — 조회 + `THIS_ONLY`**.
- [ ] **B-3. 일정에 사용자 설정 — `THIS_AND_FUTURE` / `ALL`**.
- [ ] **B-4. 일정에 팀 설정** — 조회 시 팀 userId 를 멤버로 동적 resolve.
- [ ] **B-5. 스키마 확장 (장소)**: `locations` 신규. `events` 에 `location_id` 컬럼 추가. `events_instances` 에 `status` (enum), `location_id`, `has_override_location` 컬럼 추가. CONFIRMED 전용 **partial exclusion constraint** 신규 (`EXCLUDE ... WHERE status='CONFIRMED' AND 삭제일시 IS NULL`).
- [ ] **B-6. 장소 soft-conflict 로직** — INSERT 는 `status=CONFIRMED` 로 시도 후 partial exclusion 위반 예외 catch 시 `CONFLICT` 로 재시도. 수정 시에도 동일 패턴.
- [ ] **B-7. 일정에 장소 설정** — `THIS_ONLY` / `THIS_AND_FUTURE` / `ALL` 순.
- [ ] **B-8. 단일 → 반복 전환** — 기존 인스턴스 삭제 후 rrule 기반 재생성.
